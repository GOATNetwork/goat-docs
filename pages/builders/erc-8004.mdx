# ERC-8004 for GOAT Network

ERC-8004 is an Ethereum ERC draft for **trustless agent discovery and trust signaling** through three on-chain registries: **Identity**, **Reputation**, and **Validation**.

This page summarizes the ERC, maps it to GOAT Network, and provides a practical implementation plan plus cross-chain references.

<Callout type="warning">
  ERC-8004 is currently a <strong>Draft</strong> ERC. Interfaces and semantics may evolve before finalization.
</Callout>

## Why ERC-8004 Matters for GOAT

GOAT is increasingly positioned for AI-agent-native use cases. ERC-8004 provides a standard trust layer that can complement GOAT's payment and settlement stack.

In practice:

- **Identity**: discoverable, portable on-chain agent identity
- **Reputation**: structured feedback signals from clients/users
- **Validation**: on-chain hooks for independent verification (e.g., validators/oracles/attestors)

ERC-8004 does **not** define payment rails. Payment protocols (for example x402-style flows) are complementary.

## Core Architecture (ERC-8004)

### 1) Identity Registry

- ERC-721-based identity registry (agent NFT)
- `tokenId` = `agentId`
- `tokenURI` = `agentURI` (points to agent registration file)
- Canonical global identifier model:
  - `agentRegistry = {namespace}:{chainId}:{identityRegistryAddress}`
  - `agentId = ERC-721 tokenId`

For EVM chains, namespace is typically `eip155`.

Example GOAT-format identifiers:

- Mainnet: `eip155:2345:<identityRegistryAddress>`
- Testnet3: `eip155:48816:<identityRegistryAddress>`

### 2) Reputation Registry

- Feedback as signed fixed-point values (`value`, `valueDecimals`)
- Optional tags and off-chain URI/hash references
- Supports querying and aggregation (`getSummary`, `readAllFeedback`, etc.)
- Registry-level schema standardization enables composable on-chain/off-chain scoring

### 3) Validation Registry

- Supports validation request/response workflow
- Intended for validator contracts or trust modules (e.g., re-execution, attestation)
- The wider validation part of the standard is still actively evolving

## GOAT Implementation Blueprint (Recommended)

### Phase A — Foundation

1. **Deploy singleton registries on GOAT Mainnet/Testnet3**
   - IdentityRegistry
   - ReputationRegistry
   - ValidationRegistry
2. **Publish canonical registry addresses** in docs and explorer pages.
3. **Define GOAT canonical `agentRegistry` values**:
   - `eip155:2345:<mainnetIdentityRegistry>`
   - `eip155:48816:<testnetIdentityRegistry>`

### Phase B — Developer Experience

1. Publish a **GOAT Agent Registration JSON template**.
2. Ship **SDK helpers**:
   - register agent
   - update agent URI
   - submit feedback
   - submit/read validations
3. Provide **indexing examples** (Subgraph/ETL + query APIs).

### Phase C — Trust Products

1. Add **curated reviewer sets** and weighted reputation views.
2. Add **validation classes** (basic checks, strong checks, domain-specific checks).
3. Create **risk-tiered trust profiles** (low/medium/high value operations).

## Security Considerations for GOAT

1. **Registration is not trust by itself**.
   - Identity proves ownership/control path, not behavior quality.
2. **Sybil resistance is mandatory** for production scoring.
   - Use reviewer reputation, stake, allowlists, or hybrid scoring.
3. **Prefer content-addressed or hash-anchored payloads**.
   - Keep off-chain artifacts auditable (`URI + hash`).
4. **Validation tiers should match value at risk**.
   - Light checks for low-value tasks; stronger checks for high-value actions.
